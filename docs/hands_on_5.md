# hands on 5 (チーム開発のときのテクニック)

チーム開発をする際、コードレビューは避けては通れません。


## なぜコードレビューするの？

### 品質向上

他のメンバーの目を通すことで、バグやミスを早期に発見できます。
コードの可読性や保守性を向上させるためのフィードバックを得られます。
### 知識共有

チームメンバー間でコードの意図や実装内容を共有できます。
新しい技術やベストプラクティスを学ぶ機会になります。
### 一貫性の確保

チーム全体でコーディングスタイルや設計方針を統一できます。
### リスク軽減

問題のあるコードが本番環境に入るリスクを減らせます。
### スキル向上

レビューを通じて、レビュイー（レビューを受ける側）もレビュアー（レビューする側）もスキルを向上させることができます。

---
一方で下記デメリットもあります。

### 時間がかかる
- コードレビューには時間が必要であり、開発スピードが遅くなる可能性があります。

### レビュアーの負担
- レビュアーにとって負担が大きく、他のタスクに影響を与えることがあります。


上記のようなデメリットを回避するため、レビュイーはレビューしてもらいやすいように工夫することが、大事です。


---



## 1. ブランチ戦略を決める
チームで作業する際には、ブランチ戦略を明確にしておくことが重要です。以下のような戦略がよく使われます：
- [Git Flow](https://qiita.com/KosukeSone/items/514dd24828b485c69a05): 開発、リリース、ホットフィックス用のブランチを明確に分ける。
- [GitHub Flow](https://qiita.com/tatane616/items/aec00cdc1b659761cf88#github-flow%E3%81%AE%E9%96%8B%E7%99%BA%E3%81%AE%E6%B5%81%E3%82%8C): メインブランチと機能ブランチのみを使用し、シンプルな運用を行う。
- [Trunk-Based Development](https://qiita.com/kaaaichi_i/items/9153573e224d653e6f24): メインブランチに頻繁にマージし、小さな変更を積み重ねる。

## 2. コミットメッセージを工夫する
わかりやすいコミットメッセージを記述するために、以下のルールを決めておくと良いです：
- **フォーマット例**: `[種類] 概要`
  - 種類: `feat` (新機能), `fix` (バグ修正), `docs` (ドキュメント変更) など。
  - 例: `feat: ユーザー登録機能を追加`
  - また不要なcommitがあるとレビューしずらいので、不要なcommitは削除するようにしましょう。
    ```
    git rebase -i
    ```
    📖 **参考資料**
    - [git rebase](./command_docs/rebase_and_marge.md)

## 3. コンフリクト+変更量を小さくする。
- **頻繁にリモートリポジトリを取り込む**: 作業中に`git pull`や`git fetch`を定期的に実行する。
- **小さな変更を頻繁にプッシュする**: 大きな変更を一度にプッシュするとコンフリクトが発生しやすくなります。またレビューもかなり大変になります。

## 4. PRに5w1hを意識し、どんなテストを行って自分自身はOKと判断としたか明確にする。
- **変更の意図を明確にする**: PRの説明に「何を」「なぜ」変更したのかを記載する。
- **テスト内容を明記する**:どんなテストをしたか

## 6. CI/CDの導入
- **自動テストを設定する**: PR作成時に自動でテストが実行されるようにする。
- **デプロイを自動化する**: GitHub ActionsやGitLab CI/CDを活用して、プッシュ後に自動デプロイを行う。
